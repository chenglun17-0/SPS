# 业务规则配置规范

## 1. 目的

本规范定义 SPS 当前业务配置项的来源、默认值、约束与修改流程，避免配置散落导致行为不一致。

## 2. 配置项总表

| 配置项 | 代码位置 | 默认值 | 约束/范围 | 影响模块 |
| --- | --- | --- | --- | --- |
| 项目扫描最大深度 | `src-tauri/src/commands/project.rs` -> `scan_directory(..., 3)` | `3` | 建议 `1~5` | 项目发现性能与覆盖率 |
| 扫描跳过目录 | `src-tauri/src/git/scan.rs` `SKIP_DIRS` + dot 目录规则 | `node_modules,target,.git,vendor,dist,build,.next` + 所有 `.` 开头目录（`.git` 由仓库检测特判） | 允许增补，需评估漏扫风险 | 扫描性能 |
| 工作区刷新间隔 | `src/App.tsx` `setInterval(..., 5000)` | `5000ms` | 建议 `3000~15000ms` | UI 实时性与资源占用 |
| 提交历史条数 | `src/stores/changes.ts` `getCommitLog(..., 50)` | `50` | 建议 `20~200` | 历史加载耗时 |
| 空 hunk diff 回退策略 | `src-tauri/src/git/diff.rs` `result.hunks.is_empty()` + `read_file_as_addition` | 开启（主要覆盖未跟踪文件） | 必须保留 | 新文件 diff 可见性 |
| 项目持久化文件名 | `src-tauri/src/state.rs` `projects.json` | `projects.json` | 不含敏感信息 | 启动恢复项目列表 |

## 3. 数据来源规则

1. Git 相关数据仅来自 `git2` 读取结果与本地工作区文件。
2. 项目列表持久化仅保存路径字符串，不存储 token、密码或凭据。
3. 前端展示数据必须来自 `src/lib/tauri.ts` 统一接口，不允许组件层临时拼接。

## 4. 修改流程（强制）

1. 先在本文件更新目标配置项（默认值、范围、影响评估）。
2. 再修改代码中的配置值。
3. 完成后执行最小验证：

```bash
pnpm build
cargo check --manifest-path src-tauri/Cargo.toml
```

4. 在 `tasks/todo.md` 增加回顾记录（变更原因、影响范围、回滚点）。

## 5. 回滚策略

| 场景 | 回滚方式 |
| --- | --- |
| 扫描性能下降 | 回滚 `scan_directory` 深度与 `SKIP_DIRS` 变更 |
| UI 刷新引起卡顿 | 回滚轮询间隔到 `5000ms` |
| 提交历史加载过慢 | 下调提交条数限制 |
| 持久化异常 | 回滚 `state.rs` 的持久化路径/格式改动 |

## 6. ⚠️ 严禁事项

- 严禁在多个文件中硬编码同一配置但不统一维护。
- 严禁未更新本规范直接修改配置默认值。
- 严禁引入需要网络访问或 secrets 的隐式配置。
